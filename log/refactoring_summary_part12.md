# Сводка рефакторинга (Часть 12): Реализация бизнес-логики и лимитов подписки

### Цель
Реализовать на практике проверку лимитов подписки (например, `max_waiters`) при создании новых сущностей, а также применить созданную архитектуру авторизации (RBAC) к реальному эндпоинту.

### План работ
1.  Будет создан новый эндпоинт `POST /users` для создания сотрудников.
2.  Эндпоинт будет защищен цепочкой `Middleware`: `authenticationMiddleware` и `authorizationMiddleware('user:create')`.
3.  Будет реализована бизнес-логика, которая перед созданием нового официанта или повара проверяет, не превышен ли лимит `max_waiters` для данного аккаунта.
4.  В случае превышения лимита, эндпоинт вернет ошибку `403 Forbidden`.
5.  В случае успеха, будет создан новый пользователь с хэшированным паролем.

### Ожидаемый результат
Рабочий, защищенный эндпоинт, демонстрирующий применение всей спроектированной архитектуры: аутентификации, проверки прав доступа на основе ролей и проверки бизнес-правил (лимитов подписки).
